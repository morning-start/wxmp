# 架构设计

## 设计原则

### 1. 分层架构

项目采用经典的三层架构设计：

```
┌─────────────────────────────────────┐
│         用户层 (User Layer)         │
│         用户代码 / 脚本              │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│      Spider 层 (Business Layer)     │
│    业务逻辑 / 缓存管理 / 并发控制    │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│       API 层 (Data Access Layer)    │
│    API 封装 / 请求处理 / 响应解析    │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│    微信公众平台 (External System)   │
└─────────────────────────────────────┘
```

### 2. 单一职责原则

- **API 层**: 只负责与微信公众平台交互
- **Spider 层**: 只负责业务逻辑和缓存管理
- **Tools 层**: 只提供通用工具函数

### 3. 开闭原则

- 对扩展开放：可以通过继承添加新功能
- 对修改封闭：核心 API 和工具保持稳定

### 4. 依赖倒置原则

- Spider 层依赖 API 层的抽象接口
- 用户代码依赖 Spider 层的抽象接口

## 缓存策略

### 1. FakeID 缓存

**缓存文件**: `temp/fakeids.json`

**缓存策略**: Cache Aside（旁路缓存）

应用先尝试从缓存读取数据，缓存未命中时才从数据源获取，并将数据写入缓存。

```mermaid
sequenceDiagram
    participant User as 用户
    participant Spider as Spider层
    participant Cache as 缓存文件
    participant API as API层
    participant WxMP as 微信公众平台

    User->>Spider: 请求 fakeid
    Spider->>Cache: 读取缓存
    alt 缓存命中
        Cache-->>Spider: 返回 fakeid
        Spider-->>User: 返回 fakeid
    else 缓存未命中
        Spider->>API: 搜索公众号
        API->>WxMP: 请求搜索
        WxMP-->>API: 返回搜索结果
        API-->>Spider: 返回 fakeid
        Spider->>Cache: 写入缓存
        Spider-->>User: 返回 fakeid
    end
```

### 2. 时间范围缓存

**缓存文件**: `temp/articles_info/{公众号名}.json` 和 `temp/articles_info/{公众号名}.csv`

**缓存策略**: 增量更新（基于 TimeManager）

```mermaid
sequenceDiagram
    participant User as 用户
    participant Spider as Spider层
    participant Cache as 缓存文件
    participant API as API层
    participant WxMP as 微信公众平台

    User->>Spider: 请求文章列表
    Spider->>Cache: 读取时间范围
    Cache-->>Spider: 返回缓存时间范围
    Spider->>Spider: 计算剩余时间范围
    alt 无需获取
        Spider-->>User: 返回空列表
    else 需要获取
        Spider->>API: 获取文章列表
        API->>WxMP: 请求文章列表
        WxMP-->>API: 返回文章列表
        API-->>Spider: 返回文章信息
        Spider->>Cache: 更新时间范围
        Spider-->>User: 返回文章列表
    end
```

### 3. 时间范围匹配算法

```python
def match_remaining_time_range(self, need_time: T) -> T | None:
    """
    获取剩余的时间范围

    四种情况：
    1. 完全无重叠 → 返回完整请求范围，并更新 meta 为 need_time
    2. 部分重叠（需要扩展结束） → 返回扩展部分，并更新 meta.end
    3. 部分重叠（需要扩展开始） → 返回扩展部分，并更新 meta.begin
    4. 完全覆盖 → 返回 None
    """
```

**算法逻辑详解**:

| 情况 | 条件 | 返回值 | meta 更新 |
|------|------|--------|-----------|
| 完全无重叠 | `meta.end <= need_time.begin` 或 `need_time.end <= meta.begin` | `need_time` | `meta.begin = need_time.begin`, `meta.end = need_time.end` |
| 扩展结束 | `meta.begin < need_time.begin < meta.end < need_time.end` | `(meta.end, need_time.end)` | `meta.end = need_time.end` |
| 扩展开始 | `need_time.begin < meta.begin < need_time.end < meta.end` | `(need_time.begin, meta.begin)` | `meta.begin = need_time.begin` |
| 完全覆盖 | 其他情况 | `None` | 不更新 |

## 并发设计

### 1. 文章下载并发

使用 `ThreadPoolExecutor` 实现并发下载：

```python
with ThreadPoolExecutor(max_workers=max_workers) as executor:
    futures = {
        executor.submit(TimeRangeSpider.download_article_content, task): (url, title)
        for task, url, title in tasks
    }
    for future in as_completed(futures):
        result = future.result()
```

### 2. 并发控制参数

- `max_workers`: 最大并发数（默认 5）
- `timeout`: 单个请求超时时间（默认 30 秒）
- `max_retries`: 最大重试次数（默认 3 次）

## 错误处理

### 1. 异常层次

```
Exception
├── WxMPAPIError (基础异常)
│   ├── TokenError (Token 获取失败)
│   ├── SearchBizError (搜索公众号失败)
│   └── ListExError (获取文章列表失败)
└── Exception (其他异常)
```

### 2. 错误处理策略

- **API 层**: 抛出特定异常，携带详细错误信息
- **Spider 层**: 捕获异常，记录日志，继续处理其他任务
- **用户层**: 根据业务需求决定是否继续
